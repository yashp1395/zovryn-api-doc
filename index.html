Security & RBAC

This section details the JWT authentication mechanism and visualizes the Role-Based Access Control (RBAC) matrix. Use the radar chart to compare the capabilities of different system roles.

🔑

JWT Authentication

All protected endpoints require a Bearer token in the request header.

Authorization: Bearer <token>
  • Algorithm: RSA-2048
  • Storage: Keys persist in JWT_KEY_DIRECTORY
  • Continuity: Keys are reused on restart to maintain token validity.

Role Capabilities Comparison

The system defines three strict roles. Roles are hierarchical in terms of data access but separated concerning system administration.

VIEWER Read-Only

Can view records and view the dashboard summary.

ANALYST Data Manager

Can view, create, update, and delete financial records. Can view dashboard.

ADMIN Full Access

Can manage users (create/view), manage all records, and access the dashboard.

Configuration & Specifications

Essential details for environment setup, data validation rules, and standardized error handling formats.

💻 Environment Variables

Export these variables before starting the application wrapper.

$env:DB_USERNAME='postgres'
$env:DB_PASSWORD='your_password'
$env:JWT_EXPIRATION_MS='86400000'
$env:JWT_RSA_KEY_SIZE='2048'
$env:JWT_KEY_DIRECTORY='./target/security'

Testing

Run the automated test suite using the Maven wrapper:

./mvnw.cmd test

⚠️ Validation Constraints

  • User Name not blank, valid email format, password min 8 characters.
  • Record Amount > 0, type/category not blank, notes max 500 chars.
  • Pagination page ≥ 0, size > 0.

❌ Error Handling

The API returns standard HTTP status codes (400, 401, 403, 404, 500) with a structured JSON payload:

{
  "message": "Description of the error",
  "data": null
}
